MEMORA DESTINATION - UI TEST SCENARIOS

This document contains test scenarios that can be executed by QA testers through
the Segment UI and Memora Console.

1. SETUP AND AUTHENTICATION
═══════════════════════════════════════════════════════════════════════════════

───────────────────────────────────────────────────────────────────────────────
Test Scenario 1: Successful Authentication with Valid Credentials
Test Type: Positive
Description: Verify destination can be authenticated with valid Twilio credentials
Test Steps:
  1. In Segment workspace, navigate to Destinations > Add Destination
  2. Search for "Memora" and select it
  3. Enter valid Twilio API Key in "API Key" field
  4. Enter valid Twilio API Secret in "API Secret" field
  5. Enter valid Twilio Account ID in "Twilio Account ID" field
  6. Click "Test Connection" or "Save"
Expected Results:
  - Green success message appears: "Connection successful"
  - Destination is saved and enabled
  - No error messages displayed

Test Scenario 2: Authentication with Missing API Key
Test Type: Negative
Description: Verify validation error when API Key is not provided
Test Steps:
  1. Navigate to Add Destination > Memora
  2. Leave "API Key" field empty
  3. Enter valid API Secret and Account ID
  4. Attempt to save
Expected Results:
  - Red validation error appears
  - Error message indicates "API Key is required"
  - Cannot save destination configuration

Test Scenario 3: Authentication with Missing API Secret
Test Type: Negative
Description: Verify validation error when API Secret is not provided
Test Steps:
  1. Navigate to Add Destination > Memora
  2. Enter valid API Key
  3. Leave "API Secret" field empty
  4. Enter valid Account ID
  5. Attempt to save
Expected Results:
  - Red validation error appears
  - Error message indicates "API Secret is required"
  - Cannot save destination configuration

Test Scenario 4: Authentication with Missing Twilio Account ID
Test Type: Negative
Description: Verify validation error when Account ID is not provided
Test Steps:
  1. Navigate to Add Destination > Memora
  2. Enter valid API Key and Secret
  3. Leave "Twilio Account ID" field empty
  4. Attempt to save
Expected Results:
  - Red validation error appears
  - Error message: "The root value is missing the required field 'twilioAccount'"
  - Cannot save destination configuration

Test Scenario 5: Authentication with Invalid Credentials
Test Type: Negative
Description: Verify authentication fails with incorrect credentials
Test Steps:
  1. Navigate to Add Destination > Memora
  2. Enter incorrect API Key (e.g., "invalid-key-123")
  3. Enter incorrect API Secret
  4. Enter valid Account ID
  5. Click "Test Connection"
Expected Results:
  - Error message appears: "Authentication failed" or "401 Unauthorized"
  - Destination cannot be saved
  - User prompted to check credentials

Test Scenario 6: API Secret Field Masking
Test Type: Security
Description: Verify API Secret is masked in the UI
Test Steps:
  1. Navigate to Memora destination settings
  2. Enter a value in the "API Secret" field
  3. Observe the field display
Expected Results:
  - Characters in API Secret field are displayed as dots (••••) or asterisks (***)
  - Value is not visible in plain text
  - Field type is password

Test Scenario 13: Create Profile with Email and Phone
Test Type: Positive
Description: Successfully create a profile with complete information
Preconditions:
  - Destination configured and authenticated
  - Action created with store selected

Test Data:

  - Phone: +14155551234
  - First Name: John
  - Last Name: Doe
Test Steps:
  1. From a Segment source (e.g., website, test event)
  2. Send an identify event:
     analytics.identify('user123', {
       email: 'john.doe@example.com',
       phone: '+14155551234',
       firstName: 'John',
       lastName: 'Doe'
     })

  4. Open Memora Console

  - Profile exists in Memora with email: john.doe@example.com
  - Phone is stored: +14155551234
  - firstName trait: John
  - lastName trait: Doe
  - All data correctly saved

PROFILE UPDATES

Test Scenario 20: Update Existing Profile with New Traits
Test Type: Integration
Description: Verify existing profile is updated, not duplicated
Preconditions:
  - Profile already exists with email test@example.com
  - Initial traits: { firstName: "John" }
Test Steps:
  1. Send first identify event:
     analytics.identify('user123', {
       email: 'test@example.com',
       firstName: 'John'
     })
  2. Wait for profile to be created in Memora
  3. Send second identify event with additional traits:
     analytics.identify('user123', {
       email: 'test@example.com',
       lastName: 'Doe',
       city: 'San Francisco'
     })
  4. Check Memora profile
Expected Results:
  - Only ONE profile exists with email test@example.com
  - Profile contains:
    - firstName: John (from first event, preserved)
    - lastName: Doe (added from second event)
    - city: San Francisco (added from second event)
  - Profile was updated, not duplicated

Test Scenario 21: Update Profile with Overwriting Values
Test Type: Integration
Description: Verify new values overwrite existing trait values
Preconditions:
  - Profile exists with email and city: "Boston"
Test Steps:
  1. Initial profile has city: "Boston"
  2. Send identify event:
     analytics.identify('user456', {
       email: 'user@example.com',
       city: 'New York'
     })
  3. Check profile in Memora
Expected Results:
  - city trait is updated to "New York"
  - Old value "Boston" is overwritten
  - Merge behavior updates existing values

Test Scenario 24: Batch with Sparse Data

  - Only identify events trigger the action
  - Track events don't create errors, just skipped

Test Scenario 29: Custom Subscription Filter
Test Type: Positive
Description: Verify custom FQL filters work correctly
Preconditions:
  - Custom subscription configured: type = "identify" AND traits.email EXISTS
Test Steps:
  1. Send identify event WITHOUT email:
     analytics.identify('user1', { firstName: 'NoEmail' })
  2. Send identify event WITH email:
     analytics.identify('user2', {
       email: 'has@example.com',
       firstName: 'HasEmail'
     })
  3. Check Event Delivery and profiles
Expected Results:
  - Event 1 (no email): Not processed by action, skipped
  - Event 2 (has email): Processed and profile created
  - Subscription filter correctly applied
